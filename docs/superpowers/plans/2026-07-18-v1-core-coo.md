# dafna-dev-standards v1 (core + coo) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the `dafna-dev-standards` plugin marketplace v1 — `core` (rules hook, 5 skills, 2 commands, templates) and `coo` — per `docs/superpowers/specs/2026-07-18-dev-standards-plugin-design.md`.

**Architecture:** A Claude Code plugin marketplace repo. `core` injects `RULES.md` via a SessionStart hook and ships generalized copies of proven personal/edut-app skills; `coo` ships the generalized orchestrator skill. CI validates JSON + shell. GitHub publish with protected main comes last.

**Tech Stack:** Claude Code plugin format (marketplace.json / plugin.json / hooks.json / SKILL.md / command .md), POSIX sh, GitHub Actions, `jq`, `shellcheck`.

## Global Constraints

- Repo root: `~/dev-projects/dafna-dev-standards` (work directly on `main` until Task 10 publishes; after publish, feature branches only).
- **No personal/project leakage inside `plugins/`**: the strings `Asaf`, `Nir`, `edut`, `testimony`, `Monday`, `/Users`, `~/dev-projects`, `~/Downloads` must not appear anywhere under `plugins/` (docs/ is exempt). Every task ends by running the leak gate (exit-code-safe form — a bare `&& echo LEAK || echo CLEAN` would print CLEAN on grep *errors*): `grep -rniE 'asaf|nir|edut|testimony|monday|/Users|~/dev-projects|~/Downloads' plugins/; [ $? -eq 1 ] && echo CLEAN || echo LEAK-OR-ERROR` → expect `CLEAN`.
- **Tasks 2–6 are sequential by design** (disjoint paths but one shared checkout; interleaved parallel commits are forbidden by the worktree rule). Do not parallelize them.
- All shell ships as POSIX sh (`#!/bin/sh`, no bashisms), shellcheck-clean.
- `RULES.md` hard cap: 60 lines. First line is exactly `[dafna-core rules v1.0.0]`.
- Skill frontmatter: `name` + `description` only; descriptions are triggers-only ("Use when …"), no workflow summaries, ≤1024 chars.
- Conventional commits, one commit per task minimum.
- Source materials (read-only): `~/.claude/skills/{spec-review,plan-review,_review-fundamentals,lessons-log,coo}/`, `~/dev-projects/edut-app/docs/workflows/*.md` and `~/dev-projects/edut-app/.github/pull_request_template.md` (both under the second Claude account's dev-projects tree), `plugins/core/skills/dev-env-setup/windows-wsl.md` (already shipped in this repo; originally sourced from the now-removed `docs/sources/windows-dev-environment-setup.md`).

- [ ] **Step 5: Write rules/RULES.md** (≤60 lines; content below is the complete file)

```markdown
[dafna-core rules v1.0.0]

# Shared dev-workflow rules (dafna-dev-standards/core)

## Workflow gates
- Enter plan mode for any non-trivial task (3+ steps or architectural decisions).
- Execution-mode gate: after a plan is approved, STATE the execution mode out loud
  before writing code. 3+ files or any DB migration => dispatch subagent
  implementers; the main session orchestrates, reviews, and runs gates. Inline
  implementation only for <=2-file changes.
- Feature design uses brainstorming -> spec (committed to git) -> spec-review;
  plan mode is for execution planning, not feature design. Plans get plan-review.
- Bugs: investigate -> root cause -> fix -> verify. Never guess-and-patch.
- Parallel sessions in one repo => git worktrees, never a shared checkout.

## Git
- Feature branches (feature/, fix/, docs/); conventional commit messages;
  never commit directly to main.

## CI-minute economy
- Quality gates run PRE-PR on the branch/worktree: simplify pass -> code review ->
  only then open the PR, so CI runs once on already-reviewed code.
- Batch fix commits before pushing; use rerun-failed-only on red CI.
- Docs-only PRs skip CI via paths-ignore.

## Pre-merge gates
- Simplify pass, then code review on the diff.
- Security review for diffs touching auth, PII, or public endpoints.
- No leftover debug code (console.log, debugger, stray TODO).

## Logs
- Corrections, gotchas, and validated patterns go to .claude/lessons.md
  (see the lessons-log skill). A lesson recurring across 2+ repos is promoted
  to a shared rule via PR to dafna-dev-standards.
- Significant decisions: append one line to docs/DECISIONS.md
  (format documented in that file's header; seeded by /bootstrap-repo).
```

**Interfaces:**

- [ ] **Step 1: Write scripts/validate.sh**

```sh
#!/bin/sh
# CI + local validation for the marketplace. Fails on first error.
set -e
jq . .claude-plugin/marketplace.json >/dev/null
for p in plugins/*/.claude-plugin/plugin.json; do jq . "$p" >/dev/null; done
for h in plugins/*/hooks/hooks.json; do [ -f "$h" ] && jq . "$h" >/dev/null; done
find plugins -name '*.sh' -exec shellcheck {} +
[ "$(wc -l < plugins/core/rules/RULES.md)" -le 60 ]
head -1 plugins/core/rules/RULES.md | grep -q '^\[dafna-core rules v'
for s in plugins/*/skills/*/SKILL.md; do
  head -1 "$s" | grep -qx -- '---' || { echo "missing frontmatter: $s"; exit 1; }
  grep -q '^description: Use when' "$s" || { echo "description must start 'Use when': $s"; exit 1; }
  dlen=$(grep '^description:' "$s" | head -1 | wc -c)
  [ "$dlen" -le 1024 ] || { echo "description over 1024 chars: $s"; exit 1; }
done
if grep -rniE 'asaf|nir|edut|testimony|monday|/Users|~/dev-projects|~/Downloads' plugins/; then
  echo "LEAK in plugins/"; exit 1
fi
echo OK
```
