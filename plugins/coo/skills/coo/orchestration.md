# Workflow Orchestration — working agreement

The orchestration working agreement. Governs multi-workstream / cross-repo work; the operating mode is the `/coo` skill.

- **Role split (the linchpin).** In orchestration mode the main session dispatches, synthesizes, and reports — it NEVER writes code, migrations, or scripts itself. Implementers and reviewers are ephemeral fire-and-forget subagents (isolated worktrees). A worker that needs mid-run two-way steering, watching, or user-attachability runs as its **own Claude session in a split herdr pane** (`pane split` → `pane run "claude --model <m>"`; steer via `send-text`; push-style completion via a backgrounded `herdr wait agent-status <pane> --status done|blocked`). Named teammates/SendMessage are a last resort — only when herdr is unavailable AND live steering is required; equivalently: NEVER pass `name:` on a bounded dispatch (background ≠ named — un-named background subagents still notify on completion; `name:` adds only SendMessage addressability plus idle-notification noise); they are flaky (crossed/dropped messages, silent idling) and invisible to the user. Pane caveats: unattended pane sessions can stall on permission prompts (mitigate with project allowlists or `--permission-mode` at launch + a stall-watcher), and pane ids aren't durable — re-read before each send. The orchestrator's own hands touch only: memory/tracker/docs, read-only queries, merge/deploy mechanics, human-approved operator SQL, and coordination. This keeps context clean enough to run a long multi-project arc. One task per implementer; implementers report structured facts; the orchestrator relays findings in its own words (the user never reads raw agent output).
- **Advisor reflex — scoped (2026-07-11).** An advisor subagent reviews the plan before the first state-changing edit ONLY when the plan was produced by a cheaper model or the op is irreversible/long-arc; when the orchestrator itself is a frontier-tier model (Fable/Opus) planning routine reversible work, skip it — the adversarial merge review remains the standing gate. Extends the execution-mode gate in the core plugin's RULES.md.
- **Adversarial review before every PR — not before merge.** Pre-PR sequence on the branch, all as dispatched subagents: implementer → simplify pass (code-simplifier agent; skippable on small/config diffs, say so out loud) → hostile reviewer, run on the WORKTREE BEFORE `gh pr create` (CI-minute economy: the PR opens containing already-reviewed code, so CI runs once). The reviewer verifies every claim independently against live data (don't trust the implementer's report), enumerates 5-8 specific attack surfaces, verdict CLEAR/BLOCK with file:line, severity-ranked. BLOCK → fix on the branch → delta re-review → only then open the PR. Non-blocking findings: fix small in-scope ones on the branch before the PR opens (free at this stage); tracker rows only for out-of-scope, risky, or needs-decision items. Implementer briefs must therefore end at "push the branch, do NOT open a PR" — the orchestrator opens the PR after CLEAR. The hostile reviewer deliberately replaces the built-in /code-review skill (it subsumes correctness checking and adds live verification; the skill runs inline, which the role split forbids); the simplify step replaces /simplify for the same reason. Review tiers: a single hostile reviewer subagent is the default and suffices for most PRs including engine work; escalate to a Workflow-based adversarial panel (parallel finders + refute-voting verifiers — also all subagents) only when one reviewer's blind spots are the dominant risk: irreversible data ops, security-boundary changes (auth/RLS), or a diff so broad no single context holds it. Say the tier choice out loud. Model-match: sonnet for small/config diffs, full-strength for kernel/engine/live-path; trivial config diffs may skip review — said out loud.
- **Implementer brief anatomy.** worktree ritual · incident/context with file:line · fix specified to the decision level (implementation left open) with "if you conclude X, STOP and report" tripwires · mandatory sibling sweep (grep the bug class across the repo) · live-verification requirement · gates list (build/typecheck/lint/tests/migration-lint; PR via gh, DO NOT merge) · coordination notes (concurrent siblings, migration-timestamp ranges) · standing cautions.
- **Supervised irreversible ops.** code-read verification (prove the write surface) → dry-run gate (projected delta explainable) → hard-stop predicates written into the brief, not vibes → freeze-on-failure (capture state, touch nothing; human decides recovery) → idempotent recovery paths. Fail-closed: silent-success is the only unaffordable failure. Budget a supervised shakeout window after any flip.
- **Human-in-the-loop — irreversibility is the gate.** Reversible/read-only work proceeds autonomously; prod writes, schedule changes, deletions, and scope changes come back as numbered options with one `(recommended)`. Domain questions go to the user with the data to answer; validate their domain rules against the code's actual rules. Log corrections immediately (lessons.md); never assert completion of someone else's action without confirmation.
- **Batched reporting (2026-07-16).** When several workstreams run in parallel, do NOT message the user per-task progress ("agent X idle", "step done") — those drown in the stream. Handle intermediate events (idle notifications, fix-loop rounds, watcher completions) silently or with a one-liner at most, and deliver ONE recap when a phase completes or a decision is actually needed. A message to the user must carry either a decision request or a completed-phase synthesis.
- **Watchers & continuity.** Notification wake-ups are unreliable for long jobs — use redundant monitors (the agent's own watcher + an orchestrator marker monitor) and poke stalled agents with a status summary + resume message. Verify monitors actually work. Kill stale watchers. Checkpoint at every phase boundary (tracker rows with the reviewer's exact language, lessons.md before merges, memory pickup-brief). Handoff prompts written for the next model's strength.
- **Loop taxonomy.** `/loop` `/schedule` `/goal` map to: turn-based (manual) · goal-based (iterate to a verifiable success condition) · time-based (recur) · proactive (event/schedule-triggered, no human). Reach for goal-based verification loops and proactive loops (dep-upgrades, PR triage) deliberately.
- **Extraction rule.** A pattern here graduates to its own skill only when BOTH: (1) reached for standalone outside orchestration ~3+ times, and (2) it has outgrown a rule-paragraph into a real procedure (own steps + templates + gotchas). Until then it stays inline.
